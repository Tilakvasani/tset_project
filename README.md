## 📋 Overview

**DocForge AI** is a full-stack AI document generation system. It uses **LangChain + Groq (LLaMA 3.3 70B)** to generate professional department documents and publishes them directly to a **Notion database** with full metadata tracking and **automatic version control**.

### ✨ Features

- 🤖 **AI-powered generation** via LangChain + Groq (LLaMA 3.3 70B Versatile)
- 📄 **12 document types** — NDA, Privacy Policy, SLA, Employment Contract, and more
- 🏢 **Multi-department support** — HR, Finance, Legal, Sales, IT, Operations, and more
- 📓 **One-click Notion publishing** with full metadata (title, type, department, version, word count)
- 🔢 **Automatic version control** — same Department + Doc Type → version auto-increments in Notion
- ⚡ **Redis caching** for fast repeated generation and deduplication
- 🔁 **Rate limit backoff** — handles Notion API limits gracefully
- 🌊 **Flowchart rendering** — Mermaid diagrams → PNG → Imgur → Notion image blocks
- 🎨 **Modern dark UI** built with Streamlit
- 🐳 **Docker ready** — includes `docker-compose.yml` for full stack deployment

## 🚀 Quick Start

### Prerequisites

- Python 3.11+
- Redis (optional, for caching)
- Groq API key — [console.groq.com](https://console.groq.com)
- Notion Internal Integration token — [notion.so/my-integrations](https://www.notion.so/my-integrations)
- Imgur Client ID (optional, for flowchart rendering) — [api.imgur.com/oauth2/addclient](https://api.imgur.com/oauth2/addclient)

### 1. Clone the repo

```bash
git clone https://github.com/Tilakvasani/tset_project.git
cd tset_project
```

### 2. Create virtual environment

```bash
python3.11 -m venv venv

# Windows
venv\Scripts\activate

# Mac/Linux
source venv/bin/activate
```

### 3. Install dependencies

```bash
pip install -r requirements.txt
```

### 4. Configure environment

```bash
cp .env.example .env
```

Edit `.env`:

```env
GROQ_API_KEY=gsk_your_groq_key_here
NOTION_API_KEY=ntn_your_notion_integration_token
NOTION_DATABASE_ID=your_notion_database_id
REDIS_URL=redis://localhost:6379
IMGUR_CLIENT_ID=your_imgur_client_id   # optional, for flowchart images
APP_ENV=development
LOG_LEVEL=INFO
```

### 5. Run the backend

```bash
uvicorn backend.main:app --reload
```

### 6. Run the frontend (new terminal)

```bash
streamlit run ui/streamlit_app.py
```

Open http://localhost:8501 in your browser.

## 🔢 Version Control Logic

DocForge AI implements **automatic Notion-based version control**. Every time you publish a document, the system checks if a document with the same **Department** and **Doc Type** already exists in your Notion database.

### How It Works

```
User publishes: Department = HR, Doc Type = Employee Offer Letter
        │
        ▼
_get_next_version("HR", "Employee Offer Letter")
        │
        ▼
Query Notion database:
  Filter: Department == "HR" AND Doc Type == "Employee Offer Letter"
  Sort:   Version descending
  Limit:  1 result
        │
        ├── No existing doc found  ──→  version = 1
        │
        └── Existing doc found
              existing_version = page.properties.Version.number
              version = existing_version + 1
        │
        ▼
Publish new Notion page with Version = version
```

### Version Control in Code (`notion_service.py`)

```python
async def _get_next_version(dept: str, doc_type: str) -> int:
    """
    Query Notion for existing pages with the same Department + Doc Type.
    Returns max existing version + 1, or 1 if none found.
    """
    query = {
        "filter": {
            "and": [
                {"property": "Department", "select": {"equals": dept}},
                {"property": "Doc Type",   "rich_text": {"equals": doc_type}},
            ]
        },
        "sorts": [{"property": "Version", "direction": "descending"}],
        "page_size": 1,
    }
    # ... queries Notion, returns int version number
```

### Example Scenario

| Publish # | Department | Doc Type | Version in Notion |
|-----------|-----------|----------|-------------------|
| 1st | HR | Employee Offer Letter | **v1** |
| 2nd | HR | Employee Offer Letter | **v2** |
| 3rd | HR | Employee Offer Letter | **v3** |
| 1st | Finance | Invoice Template | **v1** |
| 2nd | Finance | Invoice Template | **v2** |

Each **Department + Doc Type** combination has its own independent version counter. The version number is stored in the Notion `Version` (Number) property and also displayed inside the document's metadata callout block as `Version: v2.0`.

### What Gets Updated

When version control runs, it updates:

1. **Notion page property** — `"Version": {"number": version}`
2. **Metadata callout block** — `Version: v2.0` shown inside the Notion page
3. **API response** — `"version": 2` returned to the frontend
4. **Logs** — `Existing version found: v1 → publishing as v2`

## 🌊 Flowchart Rendering

Documents containing Mermaid flowchart blocks are automatically rendered as real images in Notion.

### Rendering Pipeline

```
Document content with ```mermaid ... ``` blocks
        │
        ▼
flowchart_renderer.py  →  mermaid_to_png_bytes()
        │
        ▼
PNG bytes uploaded to Imgur (anonymous)
        │
        ├── Upload success  →  Notion image block with Imgur URL
        │
        └── Upload fail / no IMGUR_CLIENT_ID
                         →  Numbered step-list callout (fallback)
```

Set `IMGUR_CLIENT_ID` in your `.env` to enable real image rendering. Without it, flowcharts render as readable step lists.

## ⚡ Redis Caching

All expensive operations are cached to avoid redundant LLM calls and Notion API hits.

| Cache Key | What's Cached | Invalidated When |
|-----------|---------------|-----------------|
| `departments` | All department list | Never (stable data) |
| `sections:{doc_type}` | Sections for a doc type | Never (stable data) |
| `questions:{sec_id}` | Generated questions | Never per session |
| `section_content:{sec_id}` | LLM-generated section text | On answer save or section edit |
| `notion_library` | All Notion docs (5-min TTL) | On new document published |

## 4. How the Agent Layer Works

The `agent_graph.py` wraps every `/api/rag/ask` response transparently:

```
User asks question
     ↓
rag_service.answer() runs (unchanged)
     ↓
agent_graph.run_agent() runs on the result
     ↓
if confidence == "low" AND chunks == []:
    → summarize attempted sources
    → POST to /api/agent/ticket/create
    → ticket appears in Notion DB + 🤖 Agent tab
     ↓
rag_result returned to user (unchanged — no extra text added)
```

To wire the agent into the ask route, add this to `rag_routes.py` `/ask` endpoint:

```python
@router.post("/ask")
async def api_ask(req: AskRequest):
    try:
        from backend.services.rag.rag_service import answer
        from backend.services.rag.agent_graph import run_agent   # ADD

        result = await answer(
            question=req.question,
            filters=req.filters,
            session_id=req.session_id,
            top_k=req.top_k,
            doc_a=req.doc_a,
            doc_b=req.doc_b,
        )

        # Agent layer — transparent wrap (fires ticket if low confidence)
        result = await run_agent(req.question, result, req.session_id)  # ADD

        return result
    except Exception as e:
        logger.error("Ask error: %s", e)
        raise HTTPException(status_code=500, detail=str(e))
```

## 6. Ingest Service

The existing `/api/rag/ingest` endpoint in `rag_routes.py` already calls:

```python
from backend.services.rag.ingest_service import ingest_from_notion
result = await ingest_from_notion(force=req.force)
```

The new `ingest_service.py` implements this function. It:

1. Fetches all pages from `NOTION_DATABASE_ID`
2. Extracts text block-by-block (headings, paragraphs, lists, tables, toggles)
3. Chunks text at ~800 chars with 150-char overlap (paragraph-aware)
4. Embeds in batches of 64 using `text-embedding-3-large`
5. Upserts into ChromaDB with deterministic IDs (safe to re-run)
6. Stores `{ total_docs, total_chunks, ingested_at }` in Redis

**Trigger via UI:** Streamlit → Settings → Ingest, or:
```bash
curl -X POST http://localhost:8000/api/rag/ingest -H "Content-Type: application/json" -d '{"force": true}'
```
